## 1. 隐私规则（不可协商 · NON-NEGOTIABLE）

> 违反本节任何一条都被视为**严重错误**，必须立刻停止并向用户报告。

### 1.1 唯一署名

- 全站任何位置（页面、元数据、注释、commit message、文件名、alt 文本、OG image、RSS author、sitemap）**只允许**出现 `Xiaowu` 或 `Kirito`。
- **绝不允许**出现仓库主人的真实姓名（中文或拼音、全名或单字、缩写或谐音）。

### 1.2 禁止字段清单（默认不展示）

下列信息**默认禁止**出现在任何最终交付物中：

- 真实姓名（含拼音、英文译名、谐音）
- 出生年月日 / 准确年龄
- 身份证号 / 护照号 / 任何证件号
- 手机号 / 座机号
- 家庭住址 / 详细工作地址
- 学校全名 / 院系全名（可使用「某 985」「某双一流」等模糊表述）
- 公司全名 / 团队全名（可使用「某互联网大厂」「某 AI 创业公司」等）
- 任何可被反向识别身份的组合信息

### 1.3 接收敏感信息时的处理流程

当用户提供简历、个人介绍、聊天记录等可能含敏感字段的输入时，AI **必须按以下顺序操作**：

1. **不立即写入任何文件**。
2. 列出输入中识别到的敏感字段（按 1.2 清单）。
3. 对每个字段询问用户：「展示原值 / 模糊化（提供模糊方案） / 完全隐藏」。
4. 仅在用户对**该字段**给出明确许可后，才将其写入代码或内容。
5. 写入时再次自检：是否引入了未被许可的字段？

### 1.4 敏感原始材料的存放

- 用户提供的简历原文、未脱敏材料 → 必须放入 `docs/private/`。
- `docs/private/` 已在 `.gitignore` 中，**永不提交**。
- 若用户错误地把敏感文件放在仓库其他位置，AI 必须立刻提醒并协助移动。

### 1.5 元数据同样适用

`<meta>`、Open Graph、Twitter Card、`sitemap.xml`、`rss.xml`、`package.json` 的 `author` 字段、commit author（若由 AI 触发）等元数据通道，**同样适用 1.1 与 1.2**。

## 2. 文档先行（Doc-Driven Development）

### 2.1 文档是唯一真理来源

下列文档是不同主题的**唯一真理来源**（Single Source of Truth），AI 在对应主题上的所有决策必须引用对应文档：

| 主题 | SoT 文档 |
|---|---|
| 技术架构 / 目录 / 构建 | `docs/ARCHITECTURE.md` |
| 视觉风格 / 颜色 / 字体 / 动效 | `docs/design/STYLE_GUIDE.md` |
| 设计 token | `docs/design/DESIGN_TOKENS.md` |
| 组件标准 | `docs/design/COMPONENT_PATTERNS.md` |
| 内容字段 / Schema | `docs/content/CONTENT_MODEL.md` |
| 写作规范 | `docs/content/WRITING_GUIDE.md` |
| 图片规范 / prompt | `docs/content/IMAGE_GUIDE.md` + `prompts/images.md` |
| 路线图 / 已知规划 | `docs/ROADMAP.md` |

### 2.2 RFC 流程

每个新板块、新功能、跨多文件的重构，**开发前必须**：

1. 在 `docs/rfcs/` 下基于 `0000-template.md` 新建一份 RFC，编号递增。
2. 在 RFC 中说明：背景、目标、非目标、方案、影响范围、回滚策略。
3. 等待用户在对话中确认。
4. **得到确认后**才能动手实现。
5. 完成后在 RFC 末尾标注 `Status: Implemented` 与 commit 链接。

> 例外：单文件、≤30 行、无新依赖、无视觉决策的小修可跳过 RFC，但需在 commit message 中说明。

### 2.3 文档更新责任

- 修改了视觉决策 → 同步更新 `STYLE_GUIDE.md` / `DESIGN_TOKENS.md`。
- 修改了内容字段 → 同步更新 `CONTENT_MODEL.md`。
- 新增了组件 → 同步更新 `COMPONENT_PATTERNS.md`。
- 「忘记更新文档」视为未完成的工作。

### 2.4 取舍原则（Trade-off Principles）

> 当多个方案都"看起来合理"时，按以下顺序判断。每一条都是**项目宪法**，不可被"用户图方便"或"短期省力"压倒。

#### 2.4.1 长期 > 短期

**永远向前看。** 选择对未来 1~3 年迭代更友好的方案，即使当前更费力。

不为短期痛快牺牲长期优雅。**今天的省力 = 明天的债**。

#### 2.4.2 少 > 多（增量原则）

> **宁可做少，不做多。**
> 从少加到多 < 从多减到少。

在任何"要不要做这个功能/字段/动效/装饰"的疑问中，**默认选"不做"**。

理由：
- **加上去** 永远比 **删下来** 容易（减法触动用户心理，加法不会）
- 多余功能会**养出依赖**，未来想删时面临迁移成本
- 信息密度过高违反 STYLE_GUIDE §7
- 「克制」是品味的证据，**不是缺陷**

具体应用：
- 不确定字段是否要展示 → **不展示**，schema 预留即可
- 不确定动效要不要加 → **不加**
- 不确定模块要不要做 → **不做**，写进 ROADMAP 想法池

#### 2.4.3 优雅 > 方便

**用户便利不是最高目标，对项目最优雅、最贴切才是。**

不要为了"让用户填得快""让访客点得少"而引入：
- 与项目哲学冲突的组件库
- 破坏视觉一致性的快捷方案
- 短期省事但留技术债的实现

正确的便利源自**好的架构**，而非妥协。

#### 2.4.4 一致 > 新鲜

新功能必须与现有体系**风格一致**。任何"我想试个新效果"的冲动都要回答：

> 这个新元素，3 个月后回看是否仍合适？是否会成为唯一的"装饰"破坏整体？

如果不能 100% 肯定 → **不引入**。

#### 2.4.5 取舍困境的 SOP

当陷入"难以决定"时，按这个顺序自问：

```
1. 这个决定 1 年后还重要吗？
   → 不重要：用最简方案，写进 ROADMAP 留待后续
   → 重要：进入 2

2. 现在不做，未来加上去贵不贵？
   → 不贵：现在不做（增量原则）
   → 贵：进入 3

3. 现在做了，未来删掉/改掉贵不贵？
   → 贵：放弃这个方向，找替代
   → 不贵：可以做，但写进 RFC 的 §13 Alternatives

4. 仍然犹豫 → 选「少」「克制」「向前看」的那一边
```

## 3. 风格纪律（视觉与交互）

> 完整规范见 `docs/design/STYLE_GUIDE.md`，本节摘录最高优先级红线。

### 3.1 颜色红线

- **禁止**渐变色（linear/radial/conic）作为视觉装饰。
- **禁止**霓虹色、荧光色、糖果色组合。
- **禁止**玻璃拟态（backdrop-filter blur 大面积使用，仅允许在固定导航栏轻量使用）。
- **禁止**多彩配色：除中性灰阶外，全站只允许出现**一种强调色**（当前为「克制蓝」）。
- **禁止**硬编码颜色值（`#xxxxxx` / `rgb()` / `hsl()`），所有颜色必须来自 `tokens.css` 中定义的 CSS 变量。

### 3.2 动效红线

- 时长 ≤ 300ms（标准 200ms，微交互 150ms）。
- 缓动统一使用 `cubic-bezier(0.2, 0.8, 0.2, 1)`，通过 `--ease` 引用。
- **禁止**：rotate（图标专用除外）/ skew / 弹簧 / 视差 / 跑马灯 / 自动轮播 / 过度的 hover 缩放（>1.05）。
- 必须支持 `prefers-reduced-motion: reduce`。

### 3.3 信息密度

- 留白优先于功能堆砌。**宁可少放，不可拥挤。**
- 单屏聚焦一个主信息，次要信息以折叠/分页/锚点形式提供。
- 正文阅读宽度不超过 `--prose-max`（约 680px / 65ch）。

### 3.4 圆角与阴影

- 圆角：使用 token，**禁止**非 token 数值。
- 阴影：仅在「悬浮态/激活态」使用，静态卡片优先使用 1px 边线区分。

### 3.5 图标红线

- **禁止**使用 HTML 实体 / Unicode 符号（如 `←` `→` `↗` `↓` `✕`）或 emoji 充当图标。
- **禁止**手抄 / 内联零散 SVG 路径绕过统一来源。
- 全站图标必须来自**统一图标来源**（当前为 Lucide），并通过 `<Icon>` 原子组件渲染。
- 例外：正文 / 文案中作为标点出现的符号（分隔点、破折号等）属排印标点，不受此限。
- SoT 见 `docs/design/STYLE_GUIDE.md` §9。

## 4. 工程纪律（代码层面）

### 4.1 类型与质量

- TypeScript **严格模式**（`strict: true`），**禁止** `any`（必要时使用 `unknown` + 类型守卫）。
- 提交前必须通过：`astro check` + `biome check` + `astro build`。
- 不引入未使用的导入、变量、参数。

### 4.2 依赖纪律

- 引入任何新的运行时依赖（>20KB gzip 或带运行时副作用）前，**必须**在 RFC 中论证：为什么需要、是否可以用原生方案、是否有更轻量替代。
- 优先使用 Astro / Web 平台原生能力。
- 杜绝 jQuery、Moment.js、lodash（用原生）。

### 4.3 资源纪律

- 图片必须使用 Astro 的 `<Image />` / `<Picture />`，**禁止**直接 `<img>` 引用本地图片。
- 原图放 `src/assets/`；只有当真正需要原始路径访问（favicon、robots.txt 等）时才用 `public/`。
- 字体优先系统字体栈；自定义字体必须 `font-display: swap` 且子集化。

### 4.4 路由与组件

- `pages/` 目录下文件**只做组装**，禁止写业务逻辑。
- 组件分层：`primitives` → `ui` → `sections`，**禁止跨层逆向调用**。
- 任何组件的 props 必须有 TS 类型；可选 props 必须有合理默认值。

### 4.5 国际化预留

- 用户可见文案**禁止**直接硬编码英文/中文字面量于组件中（站点结构性短语除外）。
- 应通过 `src/i18n/` 字典或 Content Collection 提供，便于将来切换语言。
- 第一阶段仅交付中文，但结构必须可扩展。

## 5. 提交与协作

### 5.1 Conventional Commits

格式：`<type>(<scope>): <subject>`

允许的 `type`：`feat` `fix` `docs` `style` `refactor` `perf` `test` `build` `ci` `chore` `revert`

示例：
- `feat(home): add hero section with featured projects`
- `docs(style-guide): clarify motion duration tokens`
- `chore(deps): bump astro to 5.x`

### 5.2 增量原则

- 每次改动聚焦一个 RFC / 一个目标。
- **禁止**「顺手大重构」。看到不顺眼的代码先记录到 `docs/ROADMAP.md` 的「想法池」，下次专门处理。
- 大型变更必须可分阶段、可回滚。

### 5.3 与用户的对话纪律

- 在做任何**有视觉/架构影响**的决策前，先口述方案、列出取舍，等用户确认。
- 当用户的指令与本文件冲突时，**先指出冲突**，再询问是否要更新本文件。
- 不替用户做未授权的取舍（尤其涉及隐私、视觉风格、依赖引入）。

## 6. 内容创作规则

### 6.1 写作风格

- 中文为主，技术术语保留英文原词（如「Hooks」「SSR」），不做生硬翻译。
- 标题、摘要、正文层级清晰；正文段落控制在 3~5 句。
- 详细规范见 `docs/content/WRITING_GUIDE.md`。

### 6.2 真实性

- 不编造项目经历、技术栈、数据指标。
- 项目卡片中的 `repo` / `demo` 链接必须真实可访问；不确定时留空，**禁止**填入示例链接。
- 文章中引用第三方观点必须注明来源。

### 6.3 AI 生成内容标注

- 整篇 AI 生成或 AI 主笔的文章，必须在 frontmatter 中标记 `aiAssisted: true`。
- 视觉素材若由 AI 生成，对应 prompt 必须记入 `prompts/images.md`。

## 7. 工作流速查（Quick Reference）

新功能 / 新板块开发流程：

```
1. 阅读 AGENTS.md（本文件）
2. 阅读相关 SoT 文档（见 §2.1）
3. 在 docs/rfcs/ 提交 RFC，等待用户确认
4. 实施变更，遵守 §3 §4 §5 §6
5. 同步更新对应 SoT 文档
6. 自检：隐私 / 风格 / 类型 / 构建
7. 按 Conventional Commits 提交
8. 在 RFC 标注 Implemented
```

收到敏感信息流程：

```
1. 不写入文件
2. 列出敏感字段
3. 逐项询问处理方式
4. 得到许可后写入
5. 自检确认未越界
```
